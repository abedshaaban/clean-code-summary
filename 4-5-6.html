What is Test-driven development?

Test-driven development is a discipline and disciplines are arbitrary behaviors that are by some substantial motive. There are 3 laws/ rules for discipline which are:

  • You are not allowed to write any production code until you have first written a test that fails because the production code doesn't exist.
  • You are not allowed to write more of a test than is sufficient to fail.
  • You are not allowed to write any more production code than is sufficient to pass the currently failing test.

This also reduces your debug time by a very large fraction. When you write code with these 3 laws, programming becomes fun, not a lot of fun but fun.

Software architecture V.S. software design

Software architecture and software design are the same. However, the goal of a software architect is to minimize the human resources required to build and maintain the software systems. To measure the design quality we simply measure the effort required to meet the needs of the customer. If the effort required to maintain a system grows with time then the design in the architecture is bad, as effort increases design quality decreases.

Software has 2 values, one is that the software does what it needs to do, and the second is that the structure of that software. The software must be changeable since the requirements of the software right now will change in the future. So if I couldn't change the software then it is useless.

How to manage a software project?

To manage any project, a project manager must choose only 3 outcomes. These outcomes come with a trade-off:

  • Good (Quality)
  • Fast (Time to market)
  • Cheap (Cost-effectiveness)
  • Done

You get to choose only three.
